## dbt run и dbt build
`dbt run`
Выполняет только SQL модели — компилирует и запускает их против базы данных в порядке DAG. Больше ничего не делает.
Что конкретно происходит внутри:

- dbt компилирует Jinja → чистый SQL
- строит каждую модель под временным именем, затем атомарно переименовывает (чтобы минимизировать даунтайм)
- соблюдает materialization каждой модели — view, table, incremental

>Что не делает: не запускает тесты, не загружает seeds, не выполняет snapshots. Если нужно — каждое из этого надо запускать отдельной командой (dbt test, dbt seed, dbt snapshot).

>Когда использовать: локальная разработка, быстрая итерация по логике модели, когда тесты сейчас не нужны.

`dbt build`
Супермножество dbt run — делает всё то же самое плюс запускает тесты, seeds и snapshots в едином DAG-порядке.
Ключевое отличие в порядке выполнения. Для каждой модели в графе:

- Загружает seeds (если есть зависимость)
- Запускает unit-тесты до материализации модели
- Материализует модель (как dbt run)
Запускает data-тесты после материализации

>Это важно: если тест на модели customers упал — dbt build не пойдёт строить зависимую модель orders. В отличие от dbt run, который просто построит обе модели без каких-либо проверок.

>Когда использовать: CI/CD, production — везде, где важна гарантия качества данных перед тем, как они пойдут дальше по графу.

## Инкрементальное обновление

Это еще один тип матереализации таблицы, котрый позволяет оптимизировать скорость загрузки, путем сокращения кол-ва строк загрузки. Мы грузим не всю таблицу, а только строки с изменениями.

> Об инкрементальном обновлении https://docs.getdbt.com/docs/build/incremental-models-overview   
Стратегии инкрементального обновления https://docs.getdbt.com/docs/build/incremental-strategy   
Конфигурирование при инкрементальном обновлении https://docs.getdbt.com/docs/build/incremental-models   
Создание нового типа материализации https://docs.getdbt.com/guides/create-new-materializations?step=1   

- Меняем тип обновления в stage слое в таблице bookings - `stg_flights__bookings.sql`

```sql
{{
    config(
        materialized='incremental',
        icremental_strategy='append'
    )
}}

select 
    book_ref,
    book_date,
    total_amount
from {{ source('demo_src', 'bookings') }}
{% if is_incremental() %}
where book_ref > (select max(book_ref) from {{ this }})
{% endif %}
```

> В конфиге меняем тип на `incremental`.   
Тип инкрементальной загрузки append (добавление)   
"{% if %} {% endif %}" - jinja шаблон   
"is_incremental()" макрос который по нескольким условиям возвращает boolen   
В блоке `where` проверяем не появлись ли новые book_ref в сырых данных   

**Какие условия должны быть выполнены, чтобы is_incremental() вернул true?**
- Модель dbt должна быть создана в БД;
- Таблица физически существует;
- Для `dbt build` не используется параметр *full-refresh*;
- У выбранной модели стоит тип материализации `incremental`;

**Инкрементальные стратегии обновления**
* `append` — новые строки просто добавляются в таблицу без проверки дублей. Если одна и та же запись пришла дважды — она вставится дважды. Подходит для иммутабельных данных (например, ивенты).

* `delete+insert` — сначала удаляет из целевой таблицы все строки по `unique_key`, затем вставляет новые. Гарантирует полную замену записи, но менее эффективен на больших данных из-за двух операций вместо одной.

* `merge` — если запись с `unique_key` уже есть в таблице — обновляет её, если нет — вставляет. Без `unique_key` ведёт себя как `append`. Дорого на больших таблицах, так как сканирует всю целевую таблицу.

* `insert_overwrite` — перезаписывает целые партиции новыми данными, не трогая остальные. Подходит для таблиц, партиционированных по дате. Не работает с данными без партиций.

* `microbatch` — разбивает данные на временны́е батчи (например, по дням или часам) и обрабатывает каждый отдельным запросом. Поддерживает параллельное выполнение. Предназначен для больших time-series датасетов.
